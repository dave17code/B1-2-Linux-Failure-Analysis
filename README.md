# 🧪 B1-2 Mission — Linux 장애 재현 및 원인 분석

## 📌 미션 소개

이 레포지토리는 **B1-2 Linux 장애 분석 미션**을 수행하기 위한 프로젝트입니다.

이번 미션의 핵심은 단순히 실행파일을 실행하는 것이 아니라,  
하나의 Linux 실행파일을 서로 다른 조건에서 실행하여 **메모리 누수, CPU 과점유, 데드락** 같은 장애를 재현하고, Linux 명령어와 로그를 통해 원인을 분석하는 것입니다.

즉, 이 미션은 다음 질문에 답하는 훈련입니다.

> “프로그램이 비정상 상태에 빠졌을 때,  
> Linux 운영체제 관점에서 무엇이 일어났는지 어떻게 증명할 수 있는가?”

---

## 🎯 미션 목표

이 미션의 목표는 다음과 같습니다.

- 동일한 실행파일을 CASE별 환경변수로 다르게 실행한다.
- 메모리, CPU, 스레드 관련 장애를 재현한다.
- 프로세스 상태, 리소스 사용량, 포트 상태, 종료 코드를 확인한다.
- 로그와 명령어 결과를 근거로 장애 원인을 분석한다.
- GitHub Issue에 CASE별 증거자료와 분석 결과를 기록한다.
- 장애 상황을 단순 실행 결과가 아니라, 운영체제 관점에서 설명할 수 있도록 정리한다.

---

## 🧱 전체 구조

이번 미션은 하나의 실행파일을 사용합니다.

```text
agent-leak-app-x86
        │
        ├── CASE 1: 메모리 제한 환경
        ├── CASE 2: CPU 과점유 환경
        └── CASE 3: 멀티스레드 데드락 환경
```

실행파일은 동일하지만, 실행할 때 전달하는 환경변수 값에 따라 다른 장애 상황이 발생합니다.

```text
MEMORY_LIMIT
CPU_MAX_OCCUPY
MULTI_THREAD_ENABLE
```

즉, CASE 1·2·3은 서로 다른 프로그램이 아니라,  
**같은 프로그램을 다른 실행 조건으로 테스트하는 구조**입니다.

---

## 🖥️ 실행 환경

이 미션은 macOS에서 OrbStack을 이용해 Ubuntu Linux 환경을 만들고 수행합니다.

```text
macOS
  ↓
OrbStack
  ↓
Ubuntu Linux
  ↓
agent-leak-app-x86 실행
  ↓
CASE별 장애 재현 및 분석
```

B1-1 미션과 동일하게 OrbStack의 Ubuntu 환경을 사용할 수 있으며, 별도의 VirtualBox 설치는 필수는 아닙니다.

---

## 🧪 CASE별 테스트 개요

| CASE | 테스트 환경 | 장애 유형 | 핵심 학습 |
|---|---|---|---|
| CASE 1 | 메모리 제한 환경 | 메모리 누수 / 강제 종료 | Memory Leak, OOM, SIGKILL, Exit Code 137 |
| CASE 2 | CPU 과점유 환경 | CPU 임계값 초과 / Watchdog 종료 | CPU Usage, Watchdog, SIGTERM, Exit Code 143 |
| CASE 3 | 멀티스레드 환경 | 데드락 / Hang | Thread, Lock, Deadlock, Blocking |

---

## 🧠 CASE 1 — 메모리 제한 환경에서의 메모리 누수 분석

### 📌 목적

CASE 1은 프로그램의 메모리 사용량이 계속 증가하다가 제한값을 초과하는 상황을 재현합니다.

이 CASE를 통해 메모리 누수, 메모리 제한, SIGKILL, 종료 코드 137의 관계를 학습합니다.

### 🧪 대표 실행 조건

```bash
MEMORY_LIMIT=60 CPU_MAX_OCCUPY=20 MULTI_THREAD_ENABLE=false ./agent-leak-app-x86
```

### 🔍 관찰 흐름

```text
프로그램 실행
   ↓
메모리 사용량 증가
   ↓
MEMORY_LIMIT 초과
   ↓
SIGKILL 발생
   ↓
종료 코드 137 확인
```

### 📚 학습 포인트

- 메모리 누수는 무엇인가?
- OOM은 메모리 누수와 어떻게 다른가?
- `SIGKILL`은 어떤 시그널인가?
- 종료 코드 `137`은 왜 발생하는가?
- 종료 코드만으로 OOM을 확정할 수 있는가?
- 프로세스의 메모리 사용량은 어떻게 확인하는가?

### ✅ 핵심 정리

> 메모리 누수는 원인이고,  
> OOM 또는 SIGKILL 종료는 결과입니다.

종료 코드 `137`은 일반적으로 `SIGKILL`에 의해 프로세스가 종료되었음을 의미합니다.  
다만 `137`이라는 결과만으로는 실제 OOM Kill인지, 프로그램 내부 로직에 의한 강제 종료인지를 단정할 수 없습니다.

따라서 CASE 1에서는 다음 증거를 함께 확인하는 것이 중요합니다.

- 실행 로그
- 메모리 증가 흐름
- 종료 코드
- 커널 로그 또는 컨테이너 상태
- SIGKILL 발생 여부

---

## 🔥 CASE 2 — CPU 과점유 및 Watchdog 종료 분석

### 📌 목적

CASE 2는 프로그램이 CPU를 과도하게 점유하는 상황을 재현하고, Watchdog이 이를 감지해 종료하는 흐름을 분석합니다.

이 CASE를 통해 CPU 사용률 증가, Watchdog 감지, SIGTERM, 종료 코드 143의 관계를 학습합니다.

### 🧪 대표 실행 조건

```bash
MEMORY_LIMIT=512 CPU_MAX_OCCUPY=80 MULTI_THREAD_ENABLE=false ./agent-leak-app-x86
```

### 🔍 관찰 흐름

```text
프로그램 실행
   ↓
CPU 사용률 증가
   ↓
CPU 임계값 초과
   ↓
Watchdog 감지
   ↓
SIGTERM 종료
```

### 📚 학습 포인트

- CPU 사용률은 어떻게 확인하는가?
- CPU 과점유 상태는 어떤 문제를 만들 수 있는가?
- Watchdog은 어떤 역할을 하는가?
- `SIGTERM`과 `SIGKILL`은 어떻게 다른가?
- 종료 코드 `143`은 무엇을 의미하는가?
- CPU 부하가 시스템 전체에 어떤 영향을 줄 수 있는가?

### ✅ 핵심 정리

> CASE 2는 CPU 과점유 상태를 감지하고,  
> 비정상 프로세스를 안전하게 종료하는 흐름을 배우는 단계입니다.

`SIGTERM`은 프로세스에게 정상 종료를 요청하는 시그널입니다.  
`SIGKILL`과 달리 프로그램이 종료 처리 로직을 수행할 기회가 있을 수 있습니다.

| 구분 | SIGTERM | SIGKILL |
|---|---|---|
| 시그널 번호 | 15 | 9 |
| 의미 | 종료 요청 | 강제 종료 |
| 프로그램 처리 가능 여부 | 가능 | 불가능 |
| 대표 종료 코드 | 143 | 137 |
| 특징 | 정상 종료 유도 | 즉시 종료 |

---

## 🧵 CASE 3 — 멀티스레드 환경에서의 데드락 분석

### 📌 목적

CASE 3은 멀티스레드 환경에서 두 스레드가 서로 자원을 기다리며 멈추는 데드락 상황을 재현합니다.

이 CASE를 통해 Thread, Lock, Deadlock, Blocking, Hang 상태를 학습합니다.

### 🧪 대표 실행 조건

```bash
MEMORY_LIMIT=512 CPU_MAX_OCCUPY=20 MULTI_THREAD_ENABLE=true ./agent-leak-app-x86
```

### 🔍 관찰 흐름

```text
Thread 1: A 자원 Lock 획득
Thread 2: B 자원 Lock 획득

Thread 1: B 자원 대기
Thread 2: A 자원 대기

결과: 서로 영원히 대기
```

### 📚 학습 포인트

- Thread는 무엇인가?
- Lock은 왜 필요한가?
- Deadlock은 어떻게 발생하는가?
- Hang 상태는 프로세스 종료와 어떻게 다른가?
- 프로세스가 살아 있어도 정상 동작이 아닐 수 있는 이유는 무엇인가?
- 멀티스레드 프로그램에서 자원 획득 순서가 왜 중요한가?

### ✅ 핵심 정리

> 프로세스가 살아 있다고 해서  
> 애플리케이션이 정상 동작 중이라는 뜻은 아닙니다.

CASE 3에서는 프로세스가 종료되지 않고 살아 있을 수 있습니다.  
하지만 내부 스레드가 서로 Lock을 기다리며 멈춰 있다면 실제 작업은 진행되지 않습니다.

즉, CASE 3의 핵심은 다음입니다.

```text
프로세스 생존 여부 ≠ 서비스 정상 동작 여부
```

## 🛠️ 주요 확인 명령어

미션 수행 중에는 다음 명령어를 사용해 증거를 수집합니다.

| 목적 | 명령어 | 설명 |
|---|---|---|
| 실행 권한 부여 | `chmod +x` | 실행파일을 실행 가능 상태로 변경 |
| 프로세스 확인 | `ps`, `pgrep` | 프로그램이 실행 중인지 확인 |
| CPU/메모리 확인 | `top`, `ps`, `free` | 리소스 사용량 확인 |
| 포트 확인 | `ss -tulnp` | 열려 있는 포트 확인 |
| 종료 코드 확인 | `echo $?` | 직전 프로세스의 종료 상태 확인 |
| 로그 확인 | `cat`, `tail`, `grep` | 실행 로그와 오류 메시지 확인 |
| 커널 로그 확인 | `dmesg`, `journalctl -k` | OOM Kill 등 커널 이벤트 확인 |
